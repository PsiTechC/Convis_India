Barge-in (Interruption) Architecture
Three layers of detection ensure users can interrupt AI speech naturally
Layer 1: Client-Side Energy Detection (Most Reliable)

During ai-speaking state, the onaudioprocess callback calculates RMS energy of incoming mic audio. If 2+ consecutive chunks exceed threshold (0.01), a "barge-in" WebSocket message is sent to server. This bypasses browser echo cancellation issues.

Layer 2: Server-Side Energy Detection

In _handle_audio_data(), the server computes RMS of incoming PCM audio during is_speaking state. If energy exceeds configured threshold, triggers interruption. Less reliable due to echo cancellation artifacts.

Layer 3: Server-Side VAD (Silero)

Silero VAD model detects speech-start transition via _on_speech_started callback. Useful for detecting actual speech vs noise. Can be suppressed by echo cancellation.

Key Insight: Browser echo cancellation can suppress user speech during AI playback, making server-side detection unreliable. Client-side detection is essential for reliable barge-in.

2. How would you fine-tune an LLM for a domain-specific task?
Decision Framework:
  1. First try prompt engineering — system prompts, few-shot examples, chain-of-thought. Often sufficient for 80% of use cases.
  2. Then try RAG — if the model needs domain-specific knowledge (documents, products, policies).
  3. Fine-tune only when you need to change the model's behavior/style/format consistently, or when prompt engineering hits token limits.
Fine-tuning Process:
  • Data Preparation: Collect 1000-10000 high-quality instruction-response pairs. Format: {"messages": [{"role": "system", ...}, {"role": "user", ...}, {"role": "assistant", ...}]}. Quality > quantity — manually review and filter.
  • Strategy Selection:
    • Full fine-tune: Only for very large datasets (100K+) and significant behavior change. Expensive (need multi-GPU).
    • LoRA (Low-Rank Adaptation): Adds small trainable matrices (rank 8-64) to attention layers. 10-100x fewer parameters. Works on single GPU.
    • QLoRA: LoRA + 4-bit quantization. Fine-tune a 70B model on a single 24GB GPU. Slight quality loss vs LoRA.
  • Hyperparameters: Learning rate: 1e-5 to 5e-5 (lower for larger models). Epochs: 2-5 (watch for overfitting). LoRA rank: 8-64 (higher = more capacity but slower). LoRA alpha: typically 2x rank.
  • Evaluation: Hold out 10-20% test set. Measure: task-specific metrics (accuracy, F1), perplexity, human evaluation. Compare against base model + prompt engineering baseline.

3. Build an enterprise chatbot that answers questions from internal documents.
Full RAG Architecture:
  1. Document Ingestion Pipeline:
    • Connectors: PDF (PyMuPDF + OCR fallback), DOCX (python-docx), XLSX, Confluence, SharePoint, Google Drive
    • Extract text, tables, images (with OCR or multimodal LLM for charts)
    • Metadata extraction: title, author, date, department, document type
  2. Chunking Strategy:
    • Recursive character splitting: 1000 chars, 200 overlap (good default)
    • Semantic chunking for higher quality: split on topic boundaries using embedding similarity
    • Preserve document structure: headings become metadata, tables kept as units
  3. Embedding & Indexing:
    • Embedding model: OpenAI text-embedding-3-small (cost-effective) or BGE/E5 (self-hosted)
    • Vector store: ChromaDB (simple, local), Pinecone (managed, scalable), Weaviate (hybrid search)
    • Store: embedding + chunk text + metadata (source, page, section)
  4. Retrieval:
    • Hybrid search: dense (vector cosine similarity) + sparse (BM25 keyword matching)
    • Top-K retrieval (k=5-10), then re-rank with cross-encoder (Cohere Rerank or BGE-reranker)
    • Metadata filtering: user permissions, document type, date range
  5. Generation:
    • System prompt: "Answer based ONLY on the provided context. If unsure, say so."
    • Inject top chunks into user message with source citations
    • Streaming response for UX
  6. API Layer: FastAPI with JWT auth, rate limiting, conversation history (sliding window), WebSocket for streaming
Key consideration: Access control — users should only retrieve documents they're authorized to see. Implement per-user metadata filtering at retrieval time.

Evaluation Metrics:
  • Automated Metrics:
    • BLEU / ROUGE: N-gram overlap with reference answers. Good for summarization, poor for open-ended generation.
    • BERTScore: Semantic similarity using BERT embeddings. Better than BLEU for paraphrases.
    • LLM-as-Judge: Use GPT-4 to evaluate outputs on criteria (relevance, accuracy, helpfulness). Score 1-5 with rubric. Surprisingly effective and scalable.
    • Faithfulness (for RAG): Does the answer only contain information from the retrieved context? Checked via NLI models or LLM-as-judge.
  • Human Evaluation: Gold standard. Use A/B testing, Likert scales, or preference ranking. Expensive but necessary for production quality.
  • Task-Specific: Accuracy (classification), Exact Match (extraction), NDCG (ranking).
Handling Hallucinations:
  • Prevention: RAG (ground responses in retrieved docs), lower temperature (0.0-0.3), explicit system prompts ("Only answer from provided context. If not found, say 'I don't have that information.'").
  • Detection: Cross-reference with source documents (NLI-based factual consistency check), self-consistency (generate multiple responses, check agreement), confidence calibration.
  • Mitigation: Source citations (force model to cite specific documents), chain-of-thought (show reasoning), human-in-the-loop for high-stakes decisions.
Tools: RAGAS (RAG evaluation), DeepEval (LLM testing framework), TruLens (observability + evaluation), LangSmith (tracing + evaluation).
5. Explain few-shot prompting, chain-of-thought, and ReAct prompting.
Few-Shot Prompting:
  • Provide 2-5 example input-output pairs in the prompt before the actual query.
  • When to use: Format standardization, classification, extraction tasks. When the model needs to understand the expected output format.
  • Tip: Choose diverse, representative examples. Order matters — most relevant example last (recency bias).
# Few-shot example Classify the sentiment: "Great product!" → Positive Classify the sentiment: "Terrible service" → Negative Classify the sentiment: "It works fine" → Neutral Classify the sentiment: "{user_input}" →
Chain-of-Thought (CoT):
  • Ask the model to "think step by step" before giving the final answer.
  • When to use: Math, logic, multi-step reasoning, complex analysis. Dramatically improves accuracy on reasoning tasks.
  • Variants: Zero-shot CoT ("Let's think step by step"), Few-shot CoT (examples with reasoning), Self-consistency (generate multiple CoT paths, take majority vote).
ReAct (Reasoning + Acting):
  • Model alternates between Thought (reasoning), Action (tool call), Observation (tool result) in a loop until it can give a final answer.
  • When to use: When the model needs external data — database lookups, API calls, web search, calculations.
  • Example flow: Thought: "I need to check the user's order status" → Action: lookup_order(id=123) → Observation: {status: "shipped"} → Thought: "The order is shipped, I can tell the user" → Final Answer.
  • Implementation: LangChain Agents, OpenAI function calling, Anthropic tool use.

7. When to choose traditional ML vs LLM-based approach for prediction/forecasting?
Choose Traditional ML When:
  • Structured/tabular data: Sales forecasting, churn prediction, fraud detection. XGBoost/LightGBM outperform LLMs on tabular data.
  • Low latency required: Sub-millisecond inference. A trained sklearn model is 1000x faster than an LLM API call.
  • Interpretability needed: Regulatory requirements (finance, healthcare). SHAP values, feature importance.
  • Large numeric datasets: Time-series forecasting with 1M+ data points. ARIMA, Prophet, LSTM.
  • Cost-sensitive: High-volume predictions (millions/day). LLM API costs would be prohibitive.
Choose LLM-Based When:
  • Unstructured text data: Sentiment analysis, document classification, summarization, Q&A.
  • Few/no training examples: Zero-shot or few-shot classification. No need to collect labeled datasets.
  • Complex reasoning: Multi-step analysis, combining text + data, generating explanations.
  • Rapid prototyping: Get a working solution in hours vs weeks of data collection + training.
  • Flexibility needed: Requirements change frequently. Modify prompt vs retrain model.
Hybrid Approach: Use LLM for feature extraction (extract entities, sentiment, intent from text) → feed as features into traditional ML model for prediction. Best of both worlds.

4. How would you design an AI system that scales from 10 to 10,000 concurrent users?

2. How do you detect and handle data drift and model degradation?
Types of Drift:
  • Data drift (covariate shift): Input distribution changes. E.g., customer queries shift from English to Spanish. Detect: KL divergence, PSI (Population Stability Index), Kolmogorov-Smirnov test on feature distributions.
  • Concept drift: Relationship between input and output changes. E.g., "positive sentiment" meaning changes over time. Detect: monitor prediction accuracy over time, sliding window performance metrics.
  • Label drift: Distribution of target variable changes. E.g., fraud rate increases from 1% to 5%.
Detection Setup:
  • Log all model inputs/outputs with timestamps.
  • Compute feature statistics daily/weekly: mean, std, min, max, cardinality.
  • Compare against baseline (training data statistics).
  • Alert on: PSI > 0.2 (significant drift), accuracy drop > 5%, p95 latency increase.
Tools: Evidently AI (open-source drift monitoring), WhyLabs, Arize, NannyML. For LLMs: LangSmith, TruLens for output quality monitoring.
Response: Investigate root cause → retrain with recent data → update feature pipeline if needed → redeploy.
3. Walk through your data cleaning and preprocessing pipeline for a messy dataset.
Systematic Pipeline:
  1. Exploratory Analysis: df.info(), df.describe(), df.isnull().sum(). Visualize distributions, identify data types, check cardinality.
  2. Handle Missing Values:
    • Numerical: median imputation (robust to outliers), or KNN imputation for correlated features.
    • Categorical: mode imputation, or create "MISSING" category if missingness is informative.
    • Drop column if >50% missing. Drop row if <5% rows affected.
  3. Outlier Detection: IQR method (1.5x IQR), Z-score (>3 std), Isolation Forest for multivariate. Decision: cap/floor (winsorize) vs remove vs keep (if domain-valid).
  4. Format Standardization: Dates to datetime, phone numbers to E.164, text to lowercase + strip whitespace, categorical to consistent labels.
  5. Deduplication: Exact duplicates: df.drop_duplicates(). Fuzzy duplicates: string similarity (Levenshtein), record linkage.
  6. Feature Engineering: Date features (day of week, hour, is_weekend), text features (length, word count, language), interaction features, aggregations.
  7. Validation: Schema validation (Great Expectations or Pandera), statistical tests (distribution checks), business rule validation.

5. Explain your approach to feature engineering for structured data.
Feature Engineering Process:
  1. Domain Understanding: Talk to domain experts. What signals do humans use to make this decision? Each insight = potential feature.
  2. Feature Categories:
    • Numerical transforms: Log transform (skewed data), binning, polynomial features, ratios (price/sqft).
    • Temporal features: Day of week, hour, month, is_holiday, time_since_last_event, rolling averages.
    • Categorical encoding: One-hot (low cardinality), target encoding (high cardinality), frequency encoding.

    • Aggregation features: Group-by statistics (mean, count, max per user/category).

  3. Feature Selection:
    • Filter methods: Correlation matrix, mutual information, chi-squared test.

    • Embedded methods: L1 regularization (built-in sparsity), tree-based feature importance.
    • SHAP values: Gold standard for understanding feature contributions. Global (which features matter overall) + local (why this specific prediction).

2. How would you build a multi-agent system for complex task collaboration?
Multi-Agent Architecture:
  • Agent Roles: Each agent has a specific system prompt defining its expertise, available tools, and decision boundaries. E.g., Researcher Agent (web search, document lookup), Analyst Agent (data processing, calculations), Writer Agent (content generation, formatting).
  • Communication Patterns:
    • Sequential (Pipeline): Agent A output → Agent B input → Agent C. Simple, predictable. Good for: research → analyze → report.
    • Hierarchical: Manager Agent delegates to Worker Agents, reviews output, iterates. Good for: complex tasks requiring quality control.
    • Collaborative: Agents share a workspace, contribute concurrently. Good for: brainstorming, multi-perspective analysis.
  • Framework Choice: LangGraph for custom flows with state persistence. CrewAI for quick role-based setup. Autogen for conversational multi-agent.
  • State Management: Shared state object (LangGraph's TypedDict) passed between agents. Each agent reads relevant state, adds its contribution. Checkpointing for recovery.
  • Error Handling: Max iterations per agent (prevent infinite loops), fallback to simpler agent if complex one fails, human escalation for uncertain decisions.
3. Explain LangGraph's state management and graph-based execution.
Core Concepts:
  • State: A TypedDict that flows through the graph. Each node reads and writes to state. E.g., {"messages": [], "context": "", "decision": ""}
  • Nodes: Functions that take state, perform an action (LLM call, tool use, data lookup), and return updated state.
  • Edges: Define flow between nodes. Can be unconditional (always go to next) or conditional (if-else routing based on state).
  • Checkpointing: State is persisted after each node. Enables: resume from failure, human-in-the-loop (pause, review, modify state, resume), time-travel debugging.
from langgraph.graph import StateGraph, END from typing import TypedDict class State(TypedDict): query: str context: list[str] answer: str graph = StateGraph(State) graph.add_node("retrieve", retrieve_docs) # fetch relevant docs graph.add_node("generate", generate_answer) # LLM generates answer graph.add_node("check", check_quality) # validate answer graph.add_edge("retrieve", "generate") graph.add_conditional_edges("check", lambda s: "end" if s["quality"] == "good" else "retrieve", {"end": END, "retrieve": "retrieve"}) # retry if bad quality app = graph.compile(checkpointer=MemorySaver())
vs LangChain Chains: Chains are linear (A → B → C). LangGraph supports loops, conditional branching, parallel execution, and persistent state. Much better for complex agent workflows.
4. LlamaIndex indexing strategies: VectorStoreIndex, TreeIndex, KeywordTableIndex.
  • VectorStoreIndex (most common): Embeds all chunks, stores in vector DB. Retrieval via cosine similarity. Best for: semantic search, open-ended Q&A. Fast retrieval, simple to set up.
  • TreeIndex: Builds a tree of summaries. Leaf nodes = document chunks. Internal nodes = LLM-generated summaries of children. Retrieval: traverse tree top-down. Best for: summarization, when you need to synthesize across many documents.
  • KeywordTableIndex: Extracts keywords from each chunk using LLM. Retrieval via keyword matching. Best for: when queries use specific terminology, structured lookup. Lower cost than vector search.
  • KnowledgeGraphIndex: Extracts entities and relationships into a graph. Best for: relational queries ("Who manages the billing team?").
  • Composable Indices: Combine multiple index types. E.g., VectorStoreIndex per document + TreeIndex over all documents for multi-level retrieval.
Decision Guide: Start with VectorStoreIndex (80% of use cases). Add KeywordTableIndex for exact-match queries. Use TreeIndex when answers require synthesizing across many documents.
5. How do you implement guardrails in an LLM application?
Guardrail Layers:
  1. Input Guardrails: Detect prompt injection (classifier or rule-based), check for PII, validate topic relevance, enforce length limits.
  2. LLM Configuration: System prompt with explicit boundaries, low temperature for factual tasks, stop sequences, max token limits.
  3. Output Guardrails: Content moderation (toxicity, bias), fact-checking against source documents, PII in output detection, format validation (JSON schema).
  4. Behavioral Guardrails: Topic boundaries (only discuss X, refuse Y), refusal patterns ("I can't help with that"), escalation triggers (route to human).
Tools:
  • NeMo Guardrails (NVIDIA): Define rails in Colang (domain-specific language). Good for: dialog management, topic control, output filtering. Integrates with LangChain.
  • Guardrails AI: Pydantic-style validators for LLM output. Good for: structured output validation, format enforcement, custom validators.
  • Custom Solution: Pre/post processing middleware. Input classifier (is this a prompt injection?) + Output classifier (is this safe to show?). Often simpler and more maintainable than framework guardrails.

2. A model is too slow/expensive in production. What optimization techniques?
Optimization Techniques (ordered by effort):
  1. Caching (low effort, high impact): Cache LLM responses for identical or similar queries. Semantic cache: embed query, if cosine similarity >0.95 with cached query, return cached response. 10-50x cost reduction for repeated queries.
  2. Prompt Optimization (low effort): Shorter system prompts (every token costs $). Remove redundant instructions. Use structured output (fewer tokens than prose).
  3. Model Downsizing (medium effort): GPT-4o → GPT-4o-mini (10x cheaper, often sufficient). Route simple queries to small model, complex to large model. Classifier decides.
  4. Quantization (medium effort): INT8: ~0% quality loss, 2x smaller, 1.5x faster. INT4 (GPTQ/AWQ): ~1-3% quality loss, 4x smaller, 2x faster. GGUF for CPU inference.
  5. Batching (medium effort): Batch multiple requests into single LLM call. Works for non-real-time tasks. OpenAI Batch API: 50% cheaper.
  6. Knowledge Distillation (high effort): Train a smaller model to mimic the large model. GPT-4 generates training data → fine-tune Llama-3-8B. 10-100x cheaper inference.
  7. Speculative Decoding (high effort): Small draft model generates candidates, large model verifies. 2-3x speedup with same quality.
  8. vLLM / TensorRT-LLM (for self-hosted): PagedAttention, continuous batching, tensor parallelism. 3-5x throughput improvement over naive serving.

4. How do you profile and optimize latency in a real-time AI pipeline?
Profiling Approach:
  • End-to-end timing: Measure total pipeline latency, then break down per component. In Convis: Audio capture (5ms) + Transport (20ms) + ASR (80ms) + LLM first token (200ms) + TTS (120ms) + Playback lead time (120ms) = ~545ms total.

Common Bottlenecks & Fixes:

  • Database queries: Connection pooling, indexing, query optimization, caching hot queries in Redis.
  • Embedding computation: Batch embeddings, cache frequent queries, use faster model (FastEmbed vs OpenAI).
  • Network round-trips: Co-locate services, reduce hops, use regional endpoints, keep-alive connections.
  • Serialization: Use msgpack or protobuf instead of JSON for large payloads. Base64 encoding for binary audio.
Key Pattern: Pipeline parallelism — don't wait for LLM to finish generating before starting TTS. Stream sentence-by-sentence. This is the single biggest latency win in Convis.

LLM Fundamentals
1. Explain the Transformer architecture: self-attention, multi-head attention, positional encoding.
Self-Attention (Scaled Dot-Product Attention):
  • Each token creates three vectors: Query (Q), Key (K), Value (V) by multiplying the input embedding with learned weight matrices.
  • Attention score = softmax(QKᵀ / √d_k) × V. The division by √d_k prevents softmax saturation.
  • Intuition: Q = "what am I looking for?", K = "what do I contain?", V = "what do I actually provide?" Score = how relevant is each other token to me.
Multi-Head Attention:
  • Run self-attention N times in parallel (N = 8 or 16 typically), each with different learned projections.
  • Concatenate outputs, project back to original dimension.
  • Why: Different heads learn different relationship types — one head might learn syntactic relationships, another semantic, another positional.
Positional Encoding:
  • Attention is permutation-invariant (no notion of order). Positional encoding adds position information.
  • Sinusoidal (original): sin/cos functions at different frequencies. Allows extrapolation to unseen lengths.
  • RoPE (Rotary Position Embedding): Used in Llama, encodes relative positions via rotation matrices. Better for long contexts.
  • ALiBi: Adds linear bias to attention scores based on distance. No learned parameters.
Full Transformer Block: Input → Multi-Head Attention → Add & LayerNorm → Feed-Forward Network (2 linear layers + ReLU/GeLU) → Add & LayerNorm → Output.

3. Explain tokenization. Compare BPE, WordPiece, SentencePiece.
  • BPE (Byte Pair Encoding): Start with characters, iteratively merge most frequent pairs. Used by GPT models. Vocabulary: 50K-100K. Handles unknown words by breaking into subwords. E.g., "unhappiness" → ["un", "happiness"] or ["un", "happ", "iness"].
  • WordPiece: Similar to BPE but uses likelihood-based merging (maximize language model probability). Used by BERT. Uses "##" prefix for subword continuations: "playing" → ["play", "##ing"].
  • SentencePiece: Language-agnostic (works on raw text, no pre-tokenization). Treats input as raw bytes. Used by Llama, T5. Handles any language without language-specific preprocessing.
Impact on Performance:
  • Vocabulary size: Larger vocab = fewer tokens per text (faster, cheaper) but larger embedding matrix. Smaller vocab = more tokens (slower, more expensive) but smaller model.
  • Multilingual: Tokenizers trained primarily on English split non-English text into many tokens (3-5x more), making non-English 3-5x more expensive.
  • Domain-specific: Medical/legal/code tokenizers can dramatically reduce token count for domain text.
4. What is the KV cache in LLM inference?
Problem: In autoregressive generation, each new token requires attending to all previous tokens. Without caching, you'd recompute attention for all previous tokens at every step — O(n²) total computation for n tokens.
Solution: Cache the Key (K) and Value (V) tensors for all previous tokens. When generating token n+1, only compute Q for the new token, reuse K and V from cache. Reduces per-step computation from O(n) to O(1).
Memory Implications:
  • KV cache size = 2 × num_layers × num_heads × head_dim × sequence_length × precision_bytes.
  • For Llama-70B at 4K context: ~2.5 GB per sequence. At 128K context: ~80 GB per sequence.
  • This is why long context models need so much GPU memory — most of it is KV cache, not model weights.
  • Optimization: PagedAttention (vLLM) manages KV cache like virtual memory pages, reducing waste from variable-length sequences. GQA (Grouped Query Attention) shares K/V across heads, reducing cache size by 4-8x.
5. Explain temperature, top-p, top-k, and frequency/presence penalty.
All these control the token sampling distribution after the model computes logits:
  • Temperature (0.0-2.0): Divides logits by temperature before softmax. Low (0.0-0.3): deterministic, focused — best for factual/code. High (0.7-1.0): creative, diverse. Temperature=0: always pick highest probability token (greedy).
  • Top-p (nucleus sampling, 0.0-1.0): Only consider tokens whose cumulative probability reaches p. Top-p=0.9: consider tokens that make up 90% of probability mass. Dynamic vocabulary size (adapts per step). Generally preferred over top-k.
  • Top-k (1-100): Only consider the k most probable tokens. Top-k=50: only sample from top 50 tokens. Fixed vocabulary size. Can be combined with top-p.
  • Frequency penalty (0.0-2.0): Penalizes tokens proportional to how many times they've appeared. Reduces repetition. Higher = more diverse vocabulary.
  • Presence penalty (0.0-2.0): Penalizes tokens that have appeared at all (binary). Encourages talking about new topics. Higher = more topic diversity.
Recommended settings: Factual/code: temp=0, top_p=1. Conversational: temp=0.7, top_p=0.9. Creative writing: temp=1.0, top_p=0.95, presence_penalty=0.5.
6. Explain RLHF. What is DPO and its advantages over PPO?
RLHF Pipeline:
  1. SFT (Supervised Fine-Tuning): Fine-tune base model on high-quality instruction-response pairs. Model learns the format and basic behavior.
  2. Reward Model Training: Collect human preferences (given two responses, which is better?). Train a reward model to predict human preference scores.
  3. PPO (Proximal Policy Optimization): Use RL to optimize the language model against the reward model. KL divergence penalty prevents the model from deviating too far from SFT model.
DPO (Direct Preference Optimization):
  • Skips the reward model entirely. Directly optimizes the language model using preference pairs.
  • Mathematically equivalent to RLHF but much simpler: just a classification loss on preferred vs rejected responses.
  • Advantages over PPO: No separate reward model needed (saves compute), no RL training loop (more stable, easier to implement), fewer hyperparameters, comparable or better results.
  • Disadvantages: Less flexible than PPO for complex reward signals, can overfit on preference data if dataset is small.
7. What are Mixture-of-Experts (MoE) models?
Architecture:
  • Replace the feed-forward layer in each Transformer block with N "expert" networks (e.g., 8 experts).
  • A gating network (router) decides which 1-2 experts to activate for each token.
  • Only active experts process the token — rest are skipped.
Why more efficient:
  • Total parameters: Very large (e.g., Mixtral 8x7B = 47B total parameters).
  • Active parameters per token: Small (e.g., Mixtral activates 2 of 8 experts = ~13B active).
  • Training cost proportional to total parameters, but inference cost proportional to active parameters.
  • Result: Model quality of a 47B model with inference speed closer to a 13B model.
Examples: Mixtral 8x7B/8x22B, GPT-4 (rumored MoE), Switch Transformer, Grok.
Challenges: Load balancing (some experts get more traffic), larger memory footprint (all experts must be in memory), training instability.
RAG Architecture
1. Explain the complete RAG pipeline and failure modes at each stage.
Pipeline & Failure Modes:
  1. Document Loading: Extract text from PDF, DOCX, HTML, etc.
    Failures: OCR errors in scanned PDFs, table structure lost, images/charts ignored, encoding issues, corrupted files.
    Mitigation: Multiple extraction methods (PyMuPDF → OCR fallback), multimodal LLM for charts, validation checks.
  2. Chunking: Split documents into manageable pieces.
    Failures: Chunks too small (lose context), too large (dilute relevance), split mid-sentence/table, important context in adjacent chunk.
    Mitigation: Recursive splitting with overlap (200 chars), sentence-boundary aware, preserve document structure.
  3. Embedding: Convert chunks to vectors.
    Failures: Embedding model doesn't understand domain jargon, dimensionality too low/high, stale embeddings after doc update.
    Mitigation: Evaluate embedding models on domain data (MTEB benchmark), re-embed on document updates.
  4. Retrieval: Find relevant chunks for a query.
    Failures: Semantic gap (query phrased differently than source), wrong chunks retrieved, too few/many results.
    Mitigation: Hybrid search (vector + keyword), query expansion/rewriting, re-ranking, metadata filtering.
  5. Generation: LLM generates answer from context.
    Failures: Hallucination (answer not in context), ignores context, "lost in the middle" (misses info in middle of long context).
    Mitigation: Strong system prompt ("ONLY answer from context"), place most relevant chunks first/last, faithfulness checking.

3. What is hybrid search? When is it better than pure vector search?
Hybrid Search = Dense (Vector) + Sparse (Keyword) retrieval:
  • Dense retrieval (vector): Semantic similarity. "How do I reset my password?" matches "Steps to change your login credentials." Misses exact terms.
  • Sparse retrieval (BM25/keyword): Exact term matching. "error code E-4052" finds documents containing that exact string. Misses semantic equivalents.
  • Hybrid: Run both, combine scores (weighted sum or Reciprocal Rank Fusion). Gets semantic understanding AND exact matching.
When hybrid is better:
  • Queries contain specific identifiers (product codes, error numbers, names)
  • Domain-specific jargon that embedding models don't understand well
  • Mixed query types (some semantic, some keyword-based)
  • Legal/medical documents where exact terminology matters
Implementation: Weaviate and Pinecone support hybrid search natively. For others: run vector search + BM25 separately, merge results with RRF (Reciprocal Rank Fusion): score = 1/(k + rank_vector) + 1/(k + rank_keyword).

2. Explain LoRA and QLoRA. How do low-rank adapters work?
LoRA (Low-Rank Adaptation):
  • Core idea: Instead of updating all model weights (billions of parameters), add small trainable matrices to specific layers. Freeze original weights.
  • How: For a weight matrix W (d×d), add ΔW = BA where B (d×r) and A (r×d), r << d. Output = Wx + BAx.
  • Rank (r): Controls adapter capacity. r=8: minimal capacity, fast. r=64: more capacity, slower. r=256: approaching full fine-tune quality.
  • Alpha: Scaling factor. Typically alpha = 2×rank. Controls how much the adapter affects output.
  • Target modules: Usually q_proj, v_proj (attention layers). Can also target k_proj, o_proj, mlp layers for more capacity.
  • Parameters: A 7B model has ~7 billion parameters. LoRA r=16 adds ~17 million trainable parameters (0.24%). Stores as a separate small file (~30MB vs 14GB for full model).
QLoRA:
  • LoRA + 4-bit quantization of the base model (NF4 data type).
  • Adapters trained in BF16, base model kept in 4-bit.
  • Result: Fine-tune a 70B model on a single 48GB GPU (vs 8x 80GB GPUs for full fine-tune).
  • Quality: ~1-2% degradation vs full LoRA, but 4-8x less memory.
3. What are AI agents? Explain the ReAct pattern.
AI Agents: LLMs that can take actions in the real world by calling tools (APIs, databases, code execution) in a loop until the task is complete.

ReAct Pattern (Reasoning + Acting):
User: "What's the weather in the city where the Convis office is located?" Thought: I need to find where Convis is located first. Action: search_database(query="Convis office location") Observation: Convis is based in Bangalore, India. Thought: Now I need to get the weather for Bangalore. Action: get_weather(city="Bangalore") Observation: Bangalore: 28°C, partly cloudy. Thought: I have all the information needed. Answer: The weather in Bangalore (where Convis is located) is 28°C and partly cloudy.
How the agent decides which tool to call:
  • Tools are described in the system prompt or via function definitions (JSON schema with name, description, parameters).
  • The LLM's training on tool-use data teaches it to match user intent to tool descriptions.
  • Key: Good tool descriptions > good tool names. Describe WHEN to use, not just WHAT it does.
Agent Loop: Receive query → Think (do I need a tool?) → If yes: call tool, get result, think again → Repeat until can answer → Return final answer. Max iterations: 5-10 to prevent infinite loops.
4. What are failure modes of AI agents? How do you handle them?
  • Infinite loops: Agent keeps calling tools without making progress. Fix: Max iteration limit (5-10), detect repeated actions, escalate to human.
  • Wrong tool selection: Agent picks search tool when it should use calculator. Fix: Better tool descriptions, few-shot examples of correct tool selection, tool-selection classifier.
  • Hallucinated parameters: Agent invents an email address or phone number to pass to a tool. Fix: Validate parameters against schema before execution, require confirmation for destructive actions.
  • Over-reliance on tools: Agent calls tools for questions it could answer directly. Fix: System prompt: "Only use tools when you don't know the answer or need real-time data."
  • Error cascading: Tool returns error, agent doesn't handle it gracefully. Fix: Return structured errors, teach agent to retry with different parameters or use alternative tool.
  • Cost explosion: Agent makes 50 API calls for a simple question. Fix: Budget limits (max tokens, max tool calls), cost monitoring per conversation.

ML Fundamentals
1. Explain bias-variance trade-off.
  • Bias: Error from overly simplistic assumptions. High bias = underfitting (model too simple, misses patterns). E.g., fitting a linear model to quadratic data.
  • Variance: Error from sensitivity to training data fluctuations. High variance = overfitting (model too complex, memorizes noise). E.g., deep tree that perfectly fits training data but fails on test.
  • Trade-off: Reducing bias (more complex model) increases variance and vice versa. Goal: find the sweet spot.
Diagnosis:
  • Underfitting: High training error AND high validation error. Fix: more features, more complex model, less regularization.
  • Overfitting: Low training error BUT high validation error (gap). Fix: more data, regularization (L1/L2/dropout), simpler model, early stopping, cross-validation.
Practical: Plot learning curves (training vs validation error vs dataset size). If they converge at high error = high bias. If large gap = high variance.
2. Compare classification metrics: accuracy, precision, recall, F1, AUC-ROC.
  • Accuracy: (TP+TN)/(Total). Misleading for imbalanced data. 99% accuracy on 1% fraud = just predicting "not fraud."
  • Precision: TP/(TP+FP). "Of predicted positives, how many are correct?" Use when FP is costly (spam detection: don't put real emails in spam).
  • Recall: TP/(TP+FN). "Of actual positives, how many did we catch?" Use when FN is costly (cancer screening: don't miss a cancer case).
  • F1: 2 × (P×R)/(P+R). Harmonic mean of precision and recall. Use when you need balance between P and R.
  • AUC-ROC: Area under ROC curve (TPR vs FPR at various thresholds). Threshold-independent. 0.5 = random, 1.0 = perfect. Good for comparing models.
  • AUC-PR: Area under Precision-Recall curve. Better than AUC-ROC for imbalanced datasets (class ratio < 1:10).
For imbalanced datasets: Use AUC-PR, F1, or F-beta (beta>1 weights recall more). Never use accuracy alone.
3. How do you handle imbalanced datasets?
Techniques (ordered by preference):
  1. Class weights: Set class_weight='balanced' in sklearn. Penalizes misclassifying minority class more. Simple, no data manipulation.
  2. Threshold tuning: Instead of default 0.5, find optimal threshold using PR curve. Often more effective than data-level techniques.
  3. SMOTE: Generate synthetic minority samples by interpolating between existing samples. Risk: can create unrealistic samples in sparse regions.
  4. Random undersampling: Remove majority class samples. Simple but loses information. Use with ensemble (BalancedRandomForest).
  5. Oversampling: Duplicate minority samples. Risk: overfitting to specific samples.
  6. Focal Loss: Modifies cross-entropy to down-weight easy examples, focus on hard ones. Great for deep learning.
  7. Ensemble methods: BalancedBagging, EasyEnsemble. Train multiple models on different balanced subsets.
Important: Always apply resampling ONLY to training data. Validation and test sets must reflect real-world distribution.

Deep Learning & NLP
1. Explain backpropagation and vanishing/exploding gradient problem.
Backpropagation:
  • Forward pass: Input → multiply by weights → apply activation → compute loss.
  • Backward pass: Compute gradient of loss w.r.t. each weight using chain rule. Start from loss, propagate backwards through each layer.
  • For each weight w: ∂Loss/∂w = ∂Loss/∂output × ∂output/∂w (chain rule through all layers).
  • Update: w = w - lr × ∂Loss/∂w. Learning rate controls step size.
Vanishing Gradient:
  • In deep networks, gradients are multiplied through many layers. If each multiplication is <1 (e.g., sigmoid activation), gradients shrink exponentially. Early layers barely learn.
  • Fix: ReLU activation (gradient = 1 for positive values), residual connections (skip connections add gradient highway), batch normalization, LSTM/GRU for sequences.
Exploding Gradient:
  • Gradients multiply to very large values. Weights blow up, loss becomes NaN.
  • Fix: Gradient clipping (torch.nn.utils.clip_grad_norm_), proper weight initialization (Xavier/He), lower learning rate.
2. Explain attention mechanism step by step.
Step-by-step (Scaled Dot-Product Attention):
  1. Create Q, K, V: For each token embedding x, compute Q = xW_Q, K = xW_K, V = xW_V. These are learned linear projections.
  2. Compute attention scores: scores = QKᵀ. This is a matrix of shape (seq_len × seq_len). Each element (i,j) = how much token i should attend to token j.
  3. Scale: scores = scores / √d_k. Prevents scores from becoming too large (which would make softmax saturate).
  4. Mask (for causal/decoder): Set future token scores to -∞ so softmax makes them 0. Token i can only attend to tokens 1..i.
  5. Softmax: attention_weights = softmax(scores). Each row sums to 1. Now represents probability distribution over which tokens to attend to.
  6. Weighted sum: output = attention_weights × V. Each token's output is a weighted combination of all Value vectors, weighted by attention.
Multi-Head: Run steps 1-6 in parallel with h different sets of W_Q, W_K, W_V (h=8 or 16). Concatenate outputs, apply final linear projection. Each head can learn different attention patterns (syntactic, semantic, positional).
3. What is knowledge distillation?
Core Idea: Train a small "student" model to mimic a large "teacher" model. Student learns from teacher's probability distributions (soft labels), not just hard labels.
  • Why soft labels? Teacher outputs [cat: 0.7, dog: 0.2, bird: 0.1]. This tells the student that "dog" is somewhat similar to "cat" — more information than hard label "cat." Called "dark knowledge."
  • Temperature: Increase teacher's temperature to soften probability distribution. Higher T = more informative soft labels.
  • Loss: α × KL_divergence(student_soft, teacher_soft) + (1-α) × CrossEntropy(student, hard_labels). Typically α = 0.7.
For LLMs:
  • Generate training data with GPT-4 → fine-tune Llama-3-8B on that data.
  • Not true distillation (no soft labels from teacher), but works well in practice.
  • Result: 8B model that performs at 50-80% of GPT-4 quality for specific tasks, at 1/100th the cost.
Real example: DistilBERT = 40% smaller, 60% faster than BERT, retains 97% of performance. Created by Hugging Face using knowledge distillation.

1. Explain Python's GIL. Threading vs multiprocessing vs asyncio.
GIL (Global Interpreter Lock):
  • A mutex that allows only one thread to execute Python bytecode at a time.
  • Why it exists: CPython's memory management (reference counting) isn't thread-safe. GIL prevents race conditions on reference counts.
  • Impact: Multi-threaded Python code doesn't truly run in parallel on multiple CPU cores for CPU-bound tasks.

3. asyncio.gather() vs create_task() vs wait()
  • asyncio.create_task(coro): Schedules a coroutine to run on the event loop. Returns immediately with a Task object. Task runs in the background. Use when you want to fire-and-forget or manage tasks individually.
  • asyncio.gather(*coros): Runs multiple coroutines concurrently and waits for ALL to complete. Returns list of results in order. Use when you need all results and they're independent.
  • asyncio.wait(tasks): More control: can wait for FIRST_COMPLETED, FIRST_EXCEPTION, or ALL_COMPLETED. Returns (done, pending) sets. Use when you need partial results or timeout handling.

Interview Q&A — Local AI
1. Why did you add a local LLM option? What's the architecture?
Answer:
  • Business need: Enterprise clients in healthcare and finance can't send conversation data to third-party APIs (GDPR, HIPAA). They need on-premise deployment with zero data exfiltration.
  • Architecture: We used Ollama which exposes an OpenAI-compatible API at localhost:11434/v1. This was key — our existing code uses the OpenAI SDK (AsyncOpenAI), so switching to Ollama required only changing the base_url. No code changes to the streaming LLM handler, sentence detection, or TTS integration.
  • Model selection: We default to Llama 3.2 (8B parameters) which runs well on a single GPU. For lower-resource environments, Mistral 7B or Gemma 2B. We built a model discovery endpoint (/api/ollama/models) that lists locally available models and a validation endpoint that resolves model name + tag mismatches.
  • Trade-off: Cloud LLMs (GPT-4o) give ~200ms TTFT with better quality. Local Ollama gives ~500ms TTFT with slightly lower quality. But $0 cost and full privacy. For many enterprise use cases, this trade-off is worth it.
2. How does the offline ASR work with VAD?
Answer:
  • Problem: Cloud ASR (Deepgram) handles streaming natively — sends interim results as you speak. Whisper is a batch model — needs complete audio segments.
  • Solution: We paired faster-whisper with Silero VAD to create pseudo-streaming:
    1. Audio chunks arrive continuously via WebSocket
    2. Silero VAD (2MB ONNX model) runs on each chunk — classifies as speech or silence
    3. Speech chunks accumulate in a buffer (max 30 seconds)
    4. When VAD detects silence after speech (300ms threshold), we send the buffered audio to faster-whisper for batch transcription
    5. Result fires the same on_transcript callback as Deepgram — downstream code is identical
  • Key design: OfflineWhisperASR implements the exact same interface as StreamingDeepgramASR. The WebRTC handler doesn't know or care which ASR is running — it just receives transcripts via callbacks.
  • Performance: Whisper "base" model with int8 quantization on CPU: ~300ms per 5-second segment. Acceptable for conversational AI where you're transcribing after the user finishes speaking.
3. How did you design the provider abstraction to make cloud/local interchangeable?
Answer — Factory + Interface Pattern:
  • Interface Contract: Each provider type (ASR, TTS, LLM) has a defined interface. ASR must provide: send_audio(), start(), close(), and fire callbacks on_transcript, on_utterance_end, on_speech_started. TTS must provide: synthesize(text) and synthesize_streaming(text, callback).
  • Factory Selection: In the handler, we read assistant.asr_provider and instantiate the right class:
    if asr_provider == "deepgram": asr = StreamingDeepgramASR(api_key, callbacks) elif asr_provider == "offline_whisper": asr = OfflineWhisperASR(callbacks) # No API key needed # Same pattern for TTS and LLM
  • Zero code changes downstream: The streaming LLM handler, WebSocket message loop, barge-in detection, and audio playback code are completely agnostic to which providers are running. This is the power of proper abstraction.
  • Fallback: If a local model fails (e.g., Ollama not running), we can detect it at initialization and fall back to cloud provider with a warning log.

1. What is Agentic AI and how is it different from RAG?
RAG: Retrieve context → Generate answer. Single-step. User asks, system answers.
Agentic AI: Autonomous multi-step problem solving. Agent plans, executes tools, evaluates results, adjusts strategy, iterates until task is complete.
  • Agentic RAG: Agent decides WHAT to search, evaluates retrieval quality, reformulates query if results are poor, synthesizes across multiple searches.
  • Key Capabilities: Planning (break complex task into steps), Tool Use (APIs, code, databases), Memory (context across steps), Self-Reflection (evaluate own output, retry).
  • In Convis: Our AI assistants are agentic — they don't just answer questions, they take actions: book meetings (calendar API), look up customer data (database integration), send emails, trigger workflows. The ReAct pattern drives tool selection during live calls.
2. What is Multimodal AI and how would you use it?
  • Definition: Models that process multiple input types: text, images, audio, video. GPT-4V, Claude 3.5, Gemini Pro Vision.
  • Use Cases:
    • RAG with images: Extract information from charts, diagrams, screenshots in documents. Use vision model instead of OCR for complex layouts.
    • Voice + Vision: "Look at this error screenshot and help me fix it" — process audio (speech) and image simultaneously.
    • Content moderation: Analyze text + images together for context-aware moderation.
  • Practical tip: For RAG with tables/charts in PDFs, sending a screenshot of the page to a vision model often works better than trying to extract text from complex layouts.
3. What is vLLM and why does it matter for self-hosted models?
  • vLLM: High-throughput LLM serving engine with PagedAttention — manages KV cache like virtual memory pages.
  • Why it matters:
    • Without vLLM: Each request pre-allocates max_tokens of KV cache. If max_tokens=4096 but response is 100 tokens, 97.5% of allocated memory is wasted.
    • With PagedAttention: KV cache allocated in small pages on demand. Memory utilization goes from ~20% to ~95%. Result: 3-5x more concurrent requests on same GPU.
    • Continuous Batching: New requests join the batch immediately, don't wait for current batch to finish. Throughput: 10-24x higher than Hugging Face Transformers.
  • When to use: Self-hosting Llama/Mistral at scale. Compare: HF Transformers (development), vLLM (production serving), TensorRT-LLM (maximum performance on NVIDIA GPUs).
4. Explain structured output / JSON mode from LLMs.
Methods (ranked by reliability):
  1. Constrained Decoding (most reliable): Force the model to only generate valid JSON tokens. Libraries: Outlines, Guidance, LMQL. 100% valid JSON guaranteed. Only works with self-hosted models.
  2. Function Calling / Tool Use (very reliable): OpenAI/Anthropic native support. Define JSON schema, model outputs structured parameters. 99%+ reliability. Works with API models.
  3. JSON Mode (reliable): response_format={"type": "json_object"}. Model guaranteed to output valid JSON. But schema not enforced — fields might be missing.
  4. Pydantic Output Parser (good): LangChain/LlamaIndex approach. Add format instructions to prompt, parse output with Pydantic, retry on failure. ~95% reliability.
  5. Prompt Engineering (least reliable): "Return your answer as JSON with fields: name, age, email." Works ~85% of the time. Need retry logic.
Best practice: Use function calling for API models. Use constrained decoding for self-hosted. Always have a retry mechanism.
5. What is Model Context Protocol (MCP)?
  • Definition: Anthropic's open standard for connecting AI models to external data sources and tools. Think of it as "USB for AI" — a universal protocol for model-to-tool communication.
  • Why it matters: Instead of building custom integrations for each tool (Jira, Slack, DB, etc.), MCP provides a standardized interface. Build one MCP server for your tool, any MCP-compatible AI can use it.
  • Architecture: MCP Client (AI app) ↔ MCP Protocol (JSON-RPC over stdio/HTTP) ↔ MCP Server (tool provider). Servers expose: Resources (data), Tools (actions), Prompts (templates).
  • Relevance to Convis: Could standardize our integration layer — instead of custom Jira/HubSpot/Email services, expose them as MCP servers. Any AI assistant could use them through a universal protocol.
Real-World Expertise (From Convis)
1. Real-time audio processing in the browser — challenges and solutions.
Challenges you solved (great for "complex problem" questions):
  • Echo cancellation suppressing user speech: Browser AEC removes sounds similar to what's playing (AI voice). When AI is speaking and user tries to interrupt, AEC suppresses the user's voice. Solution: Client-side RMS energy detection BEFORE echo cancellation, using raw mic input.
  • Audio scheduling race conditions: AudioBufferSourceNode.start(time) schedules audio for future playback. If you call stopAudioPlayback(), already-scheduled nodes still play. Solution: disconnect the GainNode — silences all scheduled sources immediately.
  • Stale async task state: User barges in, canceling the current AI response task. But the task's finally block runs and resets state, clobbering the new task. Solution: Generation ID pattern — each new task gets an incrementing ID. Finally block checks if its ID is still current before modifying state.
  • WebSocket accept timeout: Browser terminates WebSocket if server doesn't call accept() within ~30 seconds. Our server did DB lookups before accepting. Solution: accept first, then do async initialization.
  • Jitter buffer design: Network latency causes audio chunks to arrive at uneven intervals. Without buffering, playback has gaps. Solution: accumulate 150ms of audio (3600 samples @ 24kHz) before playing. Track nextPlayTime for seamless scheduling.
2. Scaling a background job scheduler for outbound campaigns.
Campaign Scheduler Design (great for system design questions):
  • Problem: Need to dial thousands of leads across multiple campaigns with concurrency limits, business hours, retry policies, and real-time status tracking.
  • Solution: Async background task with 1-second dispatch interval:
    1. Every 1 second: scan all "running" campaigns
    2. For each campaign: check timezone-aware business hours (working window)
    3. Calculate available slots: max_concurrent_lines - currently_active_calls
    4. Reserve leads from pending queue (atomic update to prevent double-dial)
    5. Dispatch via AsyncCampaignDialer (Twilio API)
    6. Stale lead detection: if a lead is "calling" for >30 seconds without a webhook, reset to "no-answer"
  • Retry strategy: Supports fixed, exponential, mixed schedule, and daily backoff. Example: attempt 1 (immediate), attempt 2 (+5 minutes), attempt 3 (next day at 10 AM).
  • Key lesson: 1-second dispatch interval gives ultra-fast call progression. With 100 concurrent lines and 30-second average call duration, system can process ~12,000 leads per hour.
3. Post-call AI processing pipeline.
Pipeline (great for "end-to-end system" questions):
  1. Recording Download: Fetch from Twilio with auth (30s timeout, 50MB max). Handle retry on network errors.
  2. Transcription: Primary: Deepgram (5-10x faster than Whisper, real-time factor <0.1). Fallback: OpenAI Whisper. Auto-selection based on API key availability.
  3. AI Analysis (using LLM): Sentiment analysis (positive/neutral/negative), call summary generation, key points extraction, action items identification.
  4. Workflow Triggers: Based on call outcome:
    • CALL_COMPLETED → Create Jira ticket with summary, update HubSpot contact, send email report
    • CALL_FAILED → Alert notification, retry scheduling
    • CAMPAIGN_COMPLETED → Generate campaign report, notify stakeholders
  5. Storage: Save transcript, sentiment, summary, recording URL to call_logs collection. Link to lead record for campaign tracking.
Key insight: This entire pipeline runs asynchronously after the call ends. User doesn't wait. Background task with retry logic ensures processing completes even if individual steps fail.
